# Skill Insights Integration — Design Document

## Background

Earlier this year, our team experimented with adding an insights feature to the Deployment Agent with the goal of generating infra plans that align more closely with how users already build and operate within Azure. In our evaluation, insights modified an average of 7.8 pre-existing resources and 26 properties based on patterns observed in existing infrastructure, with 82.9% of the generated insights being applied to the final plan. An additional 4.6 new resources and 38 properties were added on average. These results show that insights can meaningfully reduce manual editing and align new plans with the user's existing Azure environment.

Now that the infra planning experience has been brought to GitHub Copilot as an agent skill, we're integrating insights into the skill's plan generation process. This document covers the architecture for collecting and deriving insights from Azure Resource Graph data, the key design decisions around implementation approach, and how we plan to test and evaluate the feature's impact on plan quality.

## Problem Statement

The current version of the skill generates infra plans based on the user's initial prompt and uses the Azure MCP tools to get resources when necessary. Without information about the user's existing Azure environment, the skill needs to make assumptions and will use default values that align with Azure best practices. This means plans generated will likely require manual editing, as it doesn't consider naming conventions, policies etc.

## What's in Scope

- Collecting and deriving insights from the user's Azure environment
- Integrating insights into the skill's plan generation process
- Testing and evaluating the insights feature's impact on plan quality

## Architecture Deep Dive

### What Are Insights

Insights are derived from the user's existing Azure environment data. There are two types: quantitative and qualitative.

**Quantitative insights** are numerical patterns that can be aggregated, such as "top VM region" and "percentage of standard vs premium SKU". These can be deterministically computed from the ARG data.

**Qualitative insights** are architectural patterns such as resource co-occurrence within workloads (e.g., "App Service deployments are consistently paired with Key Vault, VNet, and App Insights"), security posture conventions (e.g., "production workloads use private endpoints and managed identity"), and organizational standards inferred from recurring configurations across resource groups. These need to be derived using an LLM.

For infra planning, there are more applicable qualitative insights compared to quantitative insights.

### Fetching Data from ARG

There are no existing MCP tools or APIs that provide ready-to-use insights suitable for the infra planning process, so we need to gather Azure data ourselves and summarize the data using an LLM to produce the insights.

ARG is the ideal data source because it aggregates resource metadata from all resource providers into a centralized, queryable data store. ARG supports querying resources with complex filtering, grouping and sorting via KQL, and is available across multiple subscriptions, resource groups and tenants. ARG data also removes PII and secrets like keys, which makes it safer for LLMs to handle.

The utility script queries all subscriptions accessible via the user's credential, since ARG natively supports cross-subscription queries in a single request. This gives the broadest view of the user's environment without requiring them to specify which subscriptions to include.

For large environments with many resource groups, the volume of ARG data may exceed what can fit in the subagent's context window. The script will need to truncate or prioritize data in these cases — the specific strategy for this is an open design question that will be explored during implementation.

### Deriving the Insights

We need to preprocess the ARG data before passing it to the subagent for two reasons. First is to save tokens and reduce context by removing empty configurations and instance-specific configuration. The second is to make it easier for the LLM to comprehend the data by adding things such as property aggregations.

There are two ways we can prompt the subagent:

1. **Minimal guidance** — Describe the infra planning task and user requirement but give the subagent minimal guidance on what types of insights to generate and how to generate them.
2. **Grounded guidance** — Provide a list of the types of insights that are important to infra planning, and tips on how it can derive those insights.

I recommend the grounded guidance approach as it constrains the output to known-useful insight categories, letting us focus on generating insights that are most impactful. We can compare the two in the evaluation stage.

A few additional grounding constraints worth considering:

- **Scope by resource group**, not subscription. Resource groups typically represent logical workloads, so insights derived at this level are more relevant and less noisy. That said, some organisations structure resource groups by resource type or environment rather than by workload — the subagent should account for this when deriving patterns.
- **Skip instance-specific and generic insights.** The subagent should focus on correlatable patterns — for example, prefer *"Use of managed identities + Key Vault is the recommended pattern for secretless access from Functions/Static Web Apps and for securing model/endpoint credentials"* over something like *"Managed identity used frequently"*.

### Where Should Insights Live?

The insights feature consists of two parts: getting the data and deriving insights. We need to decide (1) whether this should be an MCP tool or part of the skill, and (2) within the skill, how to implement the data fetching.

#### Decision 1: MCP Tool vs Skill

It's technically feasible to implement the entire insights process as an MCP tool — it can request ARG data and use the MCP sampling feature to generate insights by requesting the client's LLM. MCP is worth considering because it's the standard extensibility mechanism for agent tools.

However, the insights generated would be specific to infra planning and unlikely to be valuable for other tasks. The insight prompts are tightly coupled to the infra-planning context — things like resource co-occurrence patterns and security posture conventions only make sense when generating infrastructure plans. Extracting this into an MCP tool creates an abstraction that nobody else would use.

Even if we take out the LLM part and just build an MCP tool for getting the ARG data, it would essentially be a wrapper around the ARG API, and most processing would still be done via the skill.

**Decision: Implement as part of the skill**, not an MCP tool.

#### Decision 2: How to Implement Data Fetching

Within the skill, there are two choices: which *data source* to use, and how to *execute* the fetching logic.

##### Data Source: ARG REST API vs Azure CLI

There are two ways to query ARG:

1. **ARG REST API** — Provides a POST endpoint that takes in a KQL query.
2. **Azure CLI** — Allows ARG querying directly from the CLI without code setup.

ARG REST API is the better option. Requesting the REST API directly provides structured data that we can preprocess, whereas we would need to write extra code to run the Azure CLI commands and read the output from STDOUT.

**Decision: Use the ARG REST API.**

##### Execution: Utility Script vs Agent Instructions

Given that we're calling a REST API, there are two ways we can implement this in the skill:

1. **Utility script** — Create a script that the skill runs directly.
2. **Instructions** — Provide instructions on how to call the REST API and have the agent run it itself.

Utility script is the clear winner for several reasons. First, it is a first-class feature of agent skills intended for performing deterministic tasks. When run with `uv`, any required packages are automatically installed in an isolated environment. Second, the content of the script is never loaded into the agent context, which saves time and tokens (as it doesn't have to analyze it). Third, the script can be tested beforehand with unit tests. We can also explicitly define error handling. It's much less likely to error compared to option 2 as the agent doesn't need to figure out how to do it on the fly. The REST API also integrates naturally with a utility script — we can use the Azure SDK to handle both authentication and API calls in one package.

**Decision: Use a utility script that queries the ARG REST API.**

### Invocation & Parallelism

The insights collection process is designed to be independent. The skill's execution is structured in phases — Phase 1 is the research and resource discovery phase where the agent gathers documentation, WAF guidelines, and Azure best practices. Since insights only need the initial user requirement, we can instruct the main agent to spawn the insights subagent in parallel at the start of Phase 1. This means we can generate the insights while the skill completes the remaining research. At the end of Phase 1, the main agent should combine the insights and research to create the list of preliminary resources.

The insights subagent should also save the insights to a JSON file in the `.azure/` folder so that it can be reused if the user needs to iterate on the infra plan.

### Failure Modes

#### Authentication

The core dependency of the utility script is Azure SDK, which provides credentials needed to query the ARG API. Azure SDK follows a chain approach when detecting existing Azure credentials. It will check VSCode, Azure CLI, etc. for credentials.

If the user is not logged in anywhere, the skill should prompt the user to sign into Azure, and retry the script once they have confirmed they are signed in. The user should also have the option to skip sign in if they wish to bypass insights gathering. This interruption should not block the other subagents from completing Phase 1 Research.

#### Conflicting Insights

When the user's existing environment has bad practices, we don't want to inherit them. We can add an extra validation step to check insights against WAF principles and Azure best practices to identify any hard conflicts. If conflicts are found, we should present them to the user and let them decide whether to follow the existing pattern or the recommended best practice.

#### Sparse Data

When the user's Azure environment has very few resources, the ARG data may not contain enough signal to derive meaningful insights. Rather than setting a hard threshold on resource count, the subagent should only emit insights it has sufficient evidence for. If it can't produce anything useful, it returns empty and the skill proceeds with normal plan generation without insights.

## Testing

### Utility Script Testing

The utility script is deterministic code, so we can add standard unit and integration tests covering ARG data fetching, preprocessing, and error handling.

### Insight Quality Validation

We need to verify that the insights themselves are good before measuring their effect on plan quality. We can create a set of Azure environments configured with known patterns, save the ARG data, then check if the subagent produces the expected insights. The test passes if the match between expected and generated insights is above a certain threshold. We can use LLM-as-a-judge to do the comparison.

### Plan Quality

#### Evaluation Strategy

We can use A/B testing with a rubric-based evaluation using LLM-as-a-judge to assess the infra plans holistically, then compare the scores across versions. The evaluation rubric has dimensions that cover different aspects of the plan:

- **Goal Alignment** — How well the plan aligns with the user's stated goal
- **WAF Conformance** — Conformance to Azure's Well-Architected Framework best practices
- **Dependency Completeness** — Completeness of dependencies between resources
- **Deployability** — Likelihood of successful deployment

For each dimension, we write a list of criteria that needs to be met. These criteria should use either 0-5 ratings, or binary true/false scales. Research shows that 0-5 scales result in the biggest human-LLM alignment, while binary scales are most robust to noise. Ideally, each criteria would be assessed in a separate LLM call, however, due to the large number of criteria and large size of each infra plan, we will compromise by running each dimension in a new call.

For each evaluation, we run it on 30 golden prompts with 3 repeated runs per prompt to reduce noise. Each plan is graded based on the rubric, and we compute the mean and sample variance across runs. We can then use Wilcoxon's signed-rank test (α = 0.05) to determine whether any observed differences between baseline and insights-enabled scores are statistically significant. With 30 paired observations, the test has sufficient power to detect medium-to-large effects (Cohen's d ≥ 0.5). If the effect size is smaller, we may need to increase the number of golden prompts in future iterations.

Note that because insights make plans more personalised rather than objectively better, we expect the plan-quality rubric scores to remain roughly the same — any statistically significant *regression* would be a red flag, but the absence of improvement is the expected outcome. The insights-specific metrics (impact and conformance scores) are where we expect to see meaningful signal.

##### Note on Cost

Let:

- N = number of prompts (30)
- M = repeat evals per prompt (3)
- D = number of dimensions (4)
- C = concurrency (5)

Since each dimension is evaluated in a separate LLM call, the total number of calls is:

```
Total Calls = N × M + N × M × D
             = 30 × 3 + 30 × 3 × 4
             = 450 calls
```

A single plan generation takes ~8 minutes, and each dimension eval takes ~2 minutes. Running in parallel with concurrency of 5:

```
Estimated Time = (30 × 3 × 8 + 30 × 3 × 4 × 2) / 5
               = 288 minutes
               ≈ 4.8 hours
```

The baseline plans and evals only need to be generated once — we can cache them for future A/B tests. We can further reduce time and cost by stripping unnecessary content from the infra plan, and running evaluations using the OpenAI Responses API (instead of the GitHub Copilot SDK) to take advantage of prompt caching on the eval system prompt.

#### Designing the Rubric to Account for Insights

The current rubric measures plan quality based on best practices and logical correctness. Insights are not intended to make plans objectively better — they make plans more personalised and reduce manual editing by aligning with the user's existing environment. When insights conform to best practices and WAF principles, the plan quality stays the same at best — the rubric won't show an improvement because the plan was already correct without insights. For this reason, the rubric for plan correctness should be kept separate from insights evaluation.

To actually measure the impact of insights, we need a different set of criteria. There are two main areas to evaluate:

1. **Insights impact score** — Count the number of resources and properties added or modified due to insights. This produces a quantitative score that can be used for future A/B tests.
2. **Insights conformance score** — Track which insights were applied and which were not. This helps tune what types of insights to generate and how the generation process works.

Since we already have artificially crafted Azure environments for insight quality validation, we can take this a step further by generating plans for workloads that closely resemble the reference environment's workload. This gives us a more or less known answer for which insights should and shouldn't be applied, making it easier to evaluate conformance.

### Summary of Metrics

- Plan quality scores from the existing rubric (should not regress)
- Insights impact and conformance scores (new metrics)
- Token usage (insights-enabled vs baseline)
- End-to-end generation time
