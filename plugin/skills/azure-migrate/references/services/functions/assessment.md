# Assessment Phase

Generate a migration assessment report before any code changes.

## Prerequisites

- Workspace contains AWS Lambda functions, SAM templates, or CloudFormation templates
- Prompt user to upload relevant files if not present

## Assessment Steps

1. **Identify Functions** ‚Äî List all Lambda functions with runtimes, triggers, and dependencies
2. **Map AWS Services** ‚Äî Map AWS services to Azure equivalents (see [lambda-to-functions.md](services/functions/lambda-to-functions.md))
3. **Map Properties** ‚Äî Map Lambda properties to Azure Functions properties
4. **Check Dependencies** ‚Äî List 3rd-party libraries and verify Azure compatibility
5. **Analyze Code** ‚Äî Check language support and runtime differences
6. **Map Triggers** ‚Äî Identify equivalent Azure Functions triggers
7. **Map Deployment** ‚Äî Identify equivalent Azure deployment strategies (CLI, Bicep, azd)
8. **Review CI/CD** ‚Äî Check pipeline compatibility with Azure DevOps or GitHub Actions
9. **Map Monitoring** ‚Äî Map CloudWatch ‚Üí Application Insights / Azure Monitor

## Code Preview

During assessment, show a **sneak peek** of what the migrated Azure Functions code will look like for each function. Use bindings and triggers (not SDKs) in all previews, following Azure Functions best practices. **Always use the latest supported language runtime** in previews (e.g., Node.js 22, Python 3.11). This helps the user understand the migration scope before committing to code migration.

> ‚ö†Ô∏è **Binding-first rule**: Code previews MUST use `input.storageBlob()`, `output.storageBlob()`, `app.storageQueue()`, etc. instead of `BlobServiceClient`, `QueueClient`, or other SDK clients. Only use SDK for services that have no binding equivalent.

## Architecture Diagrams

Generate two diagrams:
1. **Current State** ‚Äî AWS Lambda architecture with triggers and integrations
2. **Target State** ‚Äî Azure Functions architecture showing equivalent structure
