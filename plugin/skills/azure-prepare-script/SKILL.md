# Azure Prepare (Script-Driven)

> **AUTHORITATIVE GUIDANCE — MANDATORY COMPLIANCE**
>
> This skill prepares applications for Azure deployment using a **state-machine driver
> script**. The script — not this document — owns the workflow. You **MUST** run the
> script and follow each `NEXT ACTION` block exactly. Do not improvise, reorder, or skip
> steps. When the script's instructions conflict with a security policy given to you,
> stop and ask the user for explicit confirmation.

---

## Quick Reference

| Property | Value |
|----------|-------|
| Driver script | `scripts/prepare.ps1` (PowerShell 7+, cross-platform via `pwsh`) |
| State file | `<repo>/.azure-prepare/prepare-info.json` (created by the script) |
| Plan output | `<repo>/.azure/deployment-plan.md` (generated by the script) |
| Best for | Create / modify / modernize apps and prepare them for Azure deployment |
| Next skill | **azure-validate** (then **azure-deploy**) |

## When to Use This Skill

- Create a new application, or add components to an existing one
- Modernize / migrate an existing app to Azure
- Set up Azure infrastructure (Bicep, Terraform, azure.yaml, Dockerfiles)
- Deploy to or host on Azure (preparation phase only)

This skill prepares artifacts. It does **not** deploy — `azd up`, `azd deploy`, and
`terraform apply` are handled by **azure-deploy** after **azure-validate**.

## MCP Tools

This skill orchestrates via the driver script rather than direct MCP calls. When a
`NEXT ACTION` step calls for it, prefer Azure MCP tools over raw CLI — for example
`functions_template_get` for Azure Functions templates, and the **azure-quotas** skill
for provisioning-limit checks.

## Workflow / Steps

> **Run the script in a loop. The script tells you what to do next.**

1. **Run the driver** with the repo path:

   ```bash
   pwsh scripts/prepare.ps1 -RepoPath "<absolute-path-to-repo>"
   ```

2. **Read the output block.** It is one of:
   - `NEXT ACTION` — a single step to perform. It lists what to do, which `references/`
     files to read for detail, and which `input.*` keys to fill in the state file.
   - `COMPLETE` — preparation is finished; proceed to hand-off.

3. **Do the work** the step describes (analysis, `ask_user`, scanning, generating
   artifacts, invoking related skills). The script already collected everything it could
   programmatically — review the `auto.*` fields before asking the user anything.

4. **Record results.** Edit `<repo>/.azure-prepare/prepare-info.json` and set the
   listed `input.*` keys (the script pre-creates them as `null`). Use valid JSON.

5. **Re-run the script.** It validates your input, advances as far as it can
   programmatically, and prints the next `NEXT ACTION` (or `COMPLETE`).

6. **Repeat** steps 2–5 until you receive `COMPLETE`.

7. **Hand off.** On `COMPLETE`, the script has written `.azure/deployment-plan.md` with
   status `Ready for Validation`. Invoke **azure-validate**. Do **not** run any
   deployment command directly.

> ⚠️ **Approval gate:** One step is a user-approval gate (`approval`). Present the
> generated `.azure/deployment-plan.md` to the user and only set `input.userApproved`
> to `true` after they approve.
>
> ⛔ **Never delete the user's project or workspace directory.** Destructive actions
> require `ask_user`. See [Global Rules](scripts/references/global-rules.md).

## Error Handling

| Symptom | Cause | Remediation |
|---------|-------|-------------|
| `RepoPath does not exist` | Wrong path argument | Pass the absolute workspace root path |
| Script re-asks for the same `input.*` key | Key still `null`, empty, or invalid JSON | Set a non-empty value with valid JSON, then re-run |
| `auto.azContext.available` is `false` | Not signed in to Azure CLI | Run `az login`; the `azure-context` step will prompt for subscription/location |
| Quota step blocks with `_TBD_` entries | Provisioning checklist incomplete | Invoke **azure-quotas**, complete the table, set `input.quotaChecklistMarkdown` |
| Plan file missing after `finalize-plan` | Step not reached yet | Continue the loop; the script generates it automatically before `approval` |
