# Skill Insights Integration

## Executive Summary

Earlier this year, our team experimented adding insights feature to the Deployment Agent with the goal of generating infra plans that align more closely with how users already build and operate within Azure. Internal results show that insights can provide actionable guidance for the infra planning process, helping to reduce manual editing and align new plans with users' existing Azure environments.

Now that the infra planning experience has been brought to GitHub Copilot as an agent skill, we propose integrating insights into the plan generation process to deliver a more seamless and complete experience.

## Problem Statement

The current version of the skill generates infra plans based on the user's initial prompt and uses the Azure MCP tools to get resources when necessary. Without information about the user's existing Azure environment, the skill needs to make assumptions and will use default values that align with Azure best practices. This means plans generated will likely require manual editing, as it doesn't consider naming conventions, policies etc.

## Architecture Deep Dive

### What Are Insights

Insights are derived from the user's existing Azure environment data. There are two types: quantitative and qualitative.

**Quantitative insights** are numerical patterns that can be aggregated, such as "top VM region" and "percentage of standard vs premium SKU". These can be deterministically computed from the ARG data.

**Qualitative insights** are architectural patterns such as resource co-occurrence within workloads (e.g., "App Service deployments are consistently paired with Key Vault, VNet, and App Insights"), security posture conventions (e.g., "production workloads use private endpoints and managed identity"), and organizational standards inferred from recurring configurations across resource groups. These need to be derived using an LLM.

For infra planning, there are many more useful qualitative insights compared to quantitative insights.

### Fetching Data from ARG

There are no existing MCP tools or APIs that provide ready-to-use insights suitable for the infra planning process, so we need to gather Azure data ourselves and summarize the data using an LLM to produce the insights.

ARG is the ideal data source because it aggregates resource metadata from all resource providers into a centralized, queryable data store. ARG supports querying resources with complex filtering, grouping and sorting via KQL, and is available across multiple subscriptions, resource groups and tenants. ARG data also removes PII and secrets like keys, which makes it safer for LLMs to handle.

### Deriving the Insights

We need to preprocess the ARG data before passing it to the subagent for two reasons. First is to save tokens and reduce context by removing empty configurations and instance-specific configuration. The second is to make it easier for the LLM to comprehend the data by adding things such as property aggregations.

There are two ways we can prompt the subagent:

1. **Minimal guidance** — Describe the infra planning task and user requirement but give the subagent minimal guidance on what types of insights to generate and how to generate them.
2. **Grounded guidance** — Provide a list of the types of insights that are important to infra planning, and tips on how it can derive those insights.

I am leaning towards the latter as it constrains the output to known-useful insight categories, and we can focus on generating insights that are most impactful. This is something we should test to measure any quality/time/token differences.

### Where Should Insights Live?

The insights feature consists of two parts: getting the data and deriving insights. We need to decide (1) whether this should be an MCP tool or part of the skill, and (2) within the skill, how to implement the data fetching.

#### Decision 1: MCP Tool vs Skill

It's technically feasible to implement the entire insights process as an MCP tool — it can request ARG data and use the MCP sampling feature to generate insights by requesting the client's LLM. MCP is worth considering because it's the standard extensibility mechanism for agent tools.

However, the insights generated would be specific to infra planning and unlikely to be valuable for other tasks. The insight prompts are tightly coupled to the infra-planning context — things like resource co-occurrence patterns and security posture conventions only make sense when generating infrastructure plans. Extracting this into an MCP tool creates an abstraction that nobody else would use.

Even if we take out the LLM part and just build an MCP tool for getting the ARG data, it would essentially be a wrapper around the ARG API, and most processing would still be done via the skill.

| Criteria | MCP Tool | Skill |
|---|---|---|
| Reusability | Low — insights are specific to infra planning | N/A — lives within the skill |
| Coupling | Loosely coupled but artificial separation | Naturally coupled to planning context |

| Testability | Independently testable | Testable via skill's existing test framework |

**Decision: Implement as part of the skill**, not an MCP tool.

#### Decision 2: Utility Script vs Agent Instructions

Within the skill, ARG can be queried in two ways:

1. **ARG REST API** — Provides a POST endpoint that takes in a KQL query.
2. **Azure CLI** — Allows ARG querying directly from the CLI without code setup.

And there are two ways to write the skill:

1. **Utility script** — Create a script that the skill runs directly.
2. **Instructions** — Provide instructions on how to call the ARG API or Azure CLI and have the agent run it itself.

Utility script is the clear winner for several reasons. First, it is a first-class feature of agent skills intended for performing deterministic tasks. When run with `uv`, any required packages are automatically installed in an isolated environment. Second, the content of the script is never loaded into the agent context, which saves time and tokens (as it doesn't have to analyze it). Third, the script can be tested beforehand with unit tests. We can also explicitly define error handling. It's much less likely to error compared to option 2 as the agent doesn't need to figure out how to do it on the fly.

For the data source, ARG REST API is the better option over Azure CLI. Requesting the ARG REST API directly provides structured data that we can preprocess, whereas we would need to write extra code to run the Azure CLI commands and read the output from STDOUT.

**Decision: Use a utility script that queries the ARG REST API.**

### Invocation & Parallelism

The insights collection process is designed to be independent. Since it only needs the initial user requirement, we can instruct the main agent to spawn the insights subagent in parallel at the start of Phase 1. This means we can generate the insights while the skill completes the remaining research. At the end of Phase 1, the main agent should combine the insights and research to create the list of preliminary resources.

The insights subagent should also save the insights to a JSON file in the `.azure/` folder so that it can be reused if the user needs to iterate on the infra plan.

### User Interaction Flow

### Authentication

The core dependency of the utility script is Azure SDK, which provides credentials needed to query the ARG API. Azure SDK follows a chain approach when detecting existing Azure credentials. It will check VSCode, Azure CLI, etc. for credentials.

If the user is not logged in anywhere, the skill should prompt the user to sign into Azure, and retry the script once they have confirmed they are signed in. The user should also have the option to skip sign in if they wish to bypass insights gathering. This interruption should not block the other subagents from completing Phase 1 Research.
